# 《第一次去大陸旅遊：臺灣人的一頁式生存聖經》實作計劃

> 文件用途：提供後續 AI／開發者直接執行的產品、內容、設計與技術規格。  
> 本階段交付範圍：只完成規劃，不製作網站、不建立 Git 歷史、不推送。  
> 目標讀者：第一次前往中國大陸自由行、從臺灣出發的繁體中文使用者。  
> 資料盤點日期：2026-07-11（Asia/Taipei）。所有政策、費率、航班與 App 規則在上線前必須重新查證。

## 1. 產品目標

建立一個 mobile-first、可快速掃讀的一頁式繁體中文網站。使用者應能在 5 分鐘內理解「出發前一定要完成什麼」，並能在旅途中快速找到支付、網路、交通、住宿與緊急備援資訊。

網站不是景點百科，也不是任何單一來源的改寫版；它應將 3 篇文章與 2 支影片整理成一條可行動的旅程流程，消除來源間的絕對化說法，清楚區分：

- 必備：缺少就可能無法出發或完成核心行程。
- 強烈建議：大幅降低卡關風險。
- 選配：視旅程長度、城市與個人需求決定。
- 備援：手機、網路或支付失敗時仍能繼續旅行。
- 時效資訊：需標示「最後查核日」並連結官方來源。

成功標準：讀者看完後能完成行前檢查、選定網路與支付組合、知道如何訂房與搭高鐵，並擁有至少一套離線／現金備援。

## 2. 核心內容原則

1. 全站使用臺灣繁體中文與臺灣慣用詞，例如「臺灣、行動電源、計程車、網路、App、人民幣」。搜尋關鍵字可自然保留「大陸旅遊／中國大陸自由行」。
2. 語氣務實、冷靜、友善，不製造恐慌，不使用「沒網路什麼都不能做」「一定寸步難行」等無條件斷言。
3. 每個建議盡量回答四件事：為什麼、何時做、怎麼做、失敗時怎麼辦。
4. 商業來源中的聯盟連結、旅行社銷售話術與未證實數字不得直接沿用。
5. 涉及入境、證件、金融、電信、醫療、保險、行動電源與交通規則時，部落格只作線索；正式頁面需用官方來源二次查核。
6. 頁尾清楚聲明：內容為旅遊資訊整理，不構成法律、金融或入境保證；實際規定以主管機關與服務商最新公告為準。

## 3. 一頁式資訊架構

### A. Hero：先別排景點，先搞定這 5 件事

- H1：第一次去大陸旅遊，臺灣人的一頁式生存聖經
- 副標：證件、網路、支付、住宿、交通，一頁準備到能出發。
- 首屏「五大通關卡」：台胞證、可用網路、雙支付、合格住宿、交通工具。
- 主要 CTA：「開始 30 天倒數檢查」；次要 CTA：「我已經在當地，查看緊急備援」。
- 顯示「資料最後查核：YYYY-MM-DD」。

### B. 30 天倒數 Checklist（主要互動區）

以頁籤或分段卡呈現，核取狀態存於 `localStorage`，不需登入。

#### 出發前 30–14 天

- 檢查／申辦台胞證；護照與台胞證都需攜帶。
- 檢查姓名拼寫、證件效期、機票資料一致性。
- 訂直飛或轉機航班。
- 訂住宿，確認訂房頁或客服明示可接待港澳台旅客／台胞證住客。
- 投保涵蓋中國大陸、海外醫療與行程延誤的旅平險。

#### 出發前 14–7 天

- 安裝並實名：支付寶、微信；綁定至少一張可海外交易信用卡。
- 準備第二張不同發卡機構信用卡作備援，並確認海外交易功能。
- 選擇漫遊、免翻牆 eSIM／SIM 或當地門號方案。
- 安裝高德地圖、12306 或 Trip.com；需要時安裝大眾點評／美團。
- 用台胞證完成交通與 App 可完成的驗證。

#### 出發前 3–1 天

- 小額測試支付，確認密碼、簡訊 OTP 與信用卡可用。
- 下載離線資訊：航班、飯店中英文地址、訂單、保單、緊急聯絡人。
- 準備少量人民幣現金，建議先以 300–500 元作一般備援；偏遠地區、長天數或不熟悉行動支付者自行提高。
- 確認行動電源符合當下搭機與當地規範；上線前另以官方規定核實 CCC 要求與容量限制。
- 將重要證件影本與電子備份分開存放。

#### 出發當天

- 隨身攜帶護照、台胞證、手機、充電設備、信用卡與人民幣現金。
- 在臺灣出境與大陸入境依現場標示及工作人員引導使用正確證件／通道，不寫死「本國人通道」等可能因口岸而異的稱呼。
- 抵達後先測試網路與一筆小額支付，再離開機場／車站。

### C. 證件與入境：兩張證件，各有用途

- 使用簡單對照卡說明護照與台胞證的典型使用場景。
- 台胞證應在臺灣提前辦妥；一次有效台胞證／口岸辦理只列為緊急備案，不作新手主方案。
- 「首來族」免費政策與景區優惠獨立放在「限時政策」卡：必須顯示適用期間、資格限制、官方規費不等於代辦費，並連官方來源。
- 不承諾所有口岸流程、免填表或快速通關方式一致；提示依現場指示。

### D. 網路決策器：你要的是方便、原門號收訊，還是 +86 號碼？

以三欄比較：

| 方案 | 適合誰 | 優點 | 風險／限制 |
| --- | --- | --- | --- |
| 臺灣門號漫遊 | 短天數、最怕設定 | 原號可收 OTP、操作簡單 | 價格與流量限制依電信商 |
| 免翻牆 eSIM／SIM | 手機支援、重視 Google／LINE | 通常快速、毋須換號 | 有些方案不能通話或收簡訊；品質依供應商 |
| 大陸 +86 門號 | 長住、重度使用在地服務 | 對部分 App／叫號／外送較方便 | 需辦理、短程旅遊未必必要，且不保證所有 App 都接受 |

結論不得寫成「一定要大陸門號」。建議短程新手先確保可用網路與臺灣門號能收 OTP；若行程長、需大量外送／預約或 App 明確要求 +86，再辦當地門號。

### E. 支付：主力＋副手＋實體備援

以「三層支付防線」呈現：

1. 支付寶：主要工具，出發前實名、綁卡、小額測試。
2. 微信支付：補足部分商家與小程式場景，同樣提前設定。
3. 信用卡＋人民幣現金：手機沒電、網路中斷、帳號風控或飯店押金時備援。

必須提醒：綁海外卡通常適合消費付款，不等於能轉帳、收款或儲值；手續費、免手續費門檻、交易上限與風控皆為時效資訊，實作時從支付寶／微信／中國官方指南更新，不寫死舊數字。

加入「付款失敗時」四步驟：切換支付 App → 換另一張卡 → 詢問可否現金／實體卡 → 聯絡平台或發卡銀行。不要承諾上傳證件後固定幾分鐘內必定解鎖。

### F. 必備 App 工具箱

用任務而非 App 品牌作分類，避免一長串下載清單：

- 付款／生活：支付寶、微信。
- 地圖／導航：高德地圖；說明 Google Maps 在中國大陸的資料與服務限制。
- 叫車：支付寶內滴滴小程式或滴滴 App。
- 高鐵：12306（官方、通常價格直接）或 Trip.com（介面對新手友善但可能有服務費）。
- 餐廳／景點：大眾點評、高德掃街榜；美團視需求安裝。
- 外送：美團／餓了麼，可能受門號與地址填寫限制，列為選配而非生存必備。

每張 App 卡只顯示：用途、是否必裝、是否建議在臺灣先驗證、無 +86 門號時的替代方式。

### G. 交通與住宿：最容易在現場卡住的兩關

- 住宿：訂房前取得「可接待港澳台旅客／可用台胞證登記」的明確證據，保存頁面截圖或客服文字；不把「涉外飯店」當唯一或永久有效的法律用語。
- 高鐵：比較 12306 與 Trip.com；說明台胞證實名、購票、進站與預留安檢時間。
- 城市尺度：提醒大型城市轉乘與步行時間常超出臺灣旅客直覺，行程不可排太密。
- 叫車：保存飯店中文地址，確認上車點與車牌；機場／車站優先使用官方排班或 App。

### H. 第一次去哪裡：依旅行人格選，不做硬排名

提供 6 張可橫向滑動的路線卡：

- 最穩入門：上海＋杭州＋蘇州，5–7 天。
- 歷史文化：北京或西安，5–6 天。
- 美食夜景：成都＋重慶，5–7 天。
- 海濱慢遊：青島＋煙台，4–5 天。
- 山水放空：昆明＋大理＋麗江，7–10 天。
- 自然健行：張家界／桂林／九寨溝，5–8 天。

每張卡呈現「適合誰、體力、建議天數、交通難度」，不塞入完整行程。航線資訊容易變動，只提供航空公司／機場查詢入口，不在靜態內容寫死班次。

### I. 新手最常踩的 8 個雷

1. 還沒辦妥證件就先買不可退票。
2. 住宿只看價格，沒確認能否以台胞證入住。
3. 抵達後才安裝、實名與綁卡。
4. 只準備一種支付方式。
5. 把大陸門號誤當所有短程旅客的必要條件，或反過來完全忽略 OTP／預約需求。
6. 使用 Google Maps 當唯一導航工具。
7. 低估城市尺度、安檢與轉乘時間。
8. 為免費門票繞遠路，忽略交通、住宿與時間成本。

敏感內容與當地法規提醒可用中性文字表達：「遵守當地法律與平台規範，避免傳播未核實或可能違法的內容。」不做政治說教。

### J. 緊急備援卡（可快速跳轉）

- 手機沒電：紙本／離線地址、現金、充電設備。
- 沒網路：機場／飯店 Wi-Fi、同行者熱點、離線截圖。
- 支付失敗：第二支付 App、第二張卡、現金。
- 證件遺失：先報警取得證明，再聯絡海基會緊急服務與當地出入境單位；正式聯絡資料上線前查核。
- 身體不適：保險公司海外急難救助、當地急救與同行者；緊急號碼上線前以官方資料核實。

## 4. 來源衝突處理決策

| 議題 | 來源差異 | 網站採用說法 |
| --- | --- | --- |
| 大陸門號 | GOCHINA 稱多數 App 需要且 eSIM 不能收驗證碼；Miss Xiao 主張無 +86 仍可自由行 | 短程旅客不一律需要 +86；先確保網路與 OTP，依特定 App／長住需求再辦 |

| 現金可用性 | GOCHINA 使用「現金使用率極低／大多只收行動支付」的強烈描述 | 行動支付是主流，但保留現金與實體卡備援；不宣稱現金不能用 |
| 入境通道 | 有來源寫港澳台指定通道，有來源寫本國人通道 | 依口岸現場「港澳台／台胞證」標示或詢問人員，不寫死單一通道名稱 |
| 住宿資格 | 來源普遍提醒確認接待港澳台／涉外資格，但用語不同 | 訂前直接向平台或飯店確認可持台胞證入住並保留證據 |
| 支付費率與額度 | 來源列出特定比例、門檻與上限，可能快速變動 | 上線前引用官方最新值；UI 標查核日，不把部落格數字當常數 |
| 首來族優惠 | 多來源描述證件規費與景區優惠，但資格、景區數與期限具時效性 | 獨立「限時政策」卡，只在官方查核後發布，區分官方規費、代辦費與各景區條件 |

### B. 頁面佈局與全域元件

1. **結構設計**：
   - 使用 Mobile-first 設計，斷點設為 `768px` (Tablet) 與 `1024px` (Desktop)。
   - 最大內容寬度限制為 `1200px`，兩側留有呼吸空間。
   - 頂部設有 **「毛玻璃動態導覽列 (Sticky Navigation Bar)」**：
     - `backdrop-filter: blur(12px) saturate(180%)`，背景為半透明 `--color-bg-base`。
     - 包含微型的滾動進度條 (Scroll Progress Indicator)，以 `--color-primary` 呈現。
     - 錨點選單在滑動到對應區塊時，字體由 `--color-text-secondary` 轉為 `--color-primary` 並加上硃砂小圓點。
2. **右下角快速通道**：
   - 懸浮一個名為 **「SOS 緊急備援」** 的圓形按鈕：
     - 配色為 `--color-primary`，具有向外擴散的微弱呼吸光圈 (Pulse animation)。
     - 點擊後，以彈出式抽屜 (Bottom Drawer for Mobile / Right Panel for Desktop) 瞬間展開緊急避險指南。

### C. 核心元件 UI 規格

#### 1. Hero 區：行前首屏
- **背景**：淡灰色的宣紙紋理，搭配極細的 `0.5px` 垂直線格，模擬傳統筆記本的底格。
- **左側/上方主標題**：
  - H1 使用 `--font-serif`，字重 700，字體大氣，呈現書籍扉頁的莊重感。
  - 「資料最後查核」以硃砂紅「印章」風格框線標示 (紅色圓角邊框，帶有輕微的 `rotate(-3deg)` 傾斜，模擬手動蓋印)。
- **行前 5 大通關卡 (Pass Cards)**：
  - 五張橫向排列的迷你卡片 (台胞證、網路、支付、住宿、交通)。
  - 卡片預設為淺灰色邊框，當滑鼠懸停 (hover) 或點擊時，利用 `transform: translateY(-8px) scale(1.02)` 浮起，並顯示相應的 icon 與一秒的微光掃過效果 (Shimmer effect)。
  - 右下方 CTA 「開始 30 天檢查」：大型實心硃砂紅按鈕，點擊時伴隨輕微按壓縮小回饋。

#### 2. 30 天倒數 Checklist（核心互動）
- **介面佈局**：
  - 頂部為時間軸進度環，顯示「已完成 X / Y 項」，環狀進度條採用竹青綠 SVG 動態繪製。
  - 四個時間段 (30天、14天、3天、當天) 採用「手風琴分段卡 (Accordion Sections)」，點擊可平滑展開/收摺。
- **互動微動效**：
  - Checklist 項目的 Checkbox 為客製化設計。當點擊勾選時：
    - Checkbox 內部以 SVG 劃出一個竹青綠的勾號。
    - 卡片背景從純白平滑過渡到 `--color-success-light`，邊框變為竹青綠。
    - 內容文字加上 `--color-text-muted`，並以淡雅的 `--color-success` 刪除線平滑畫過。
    - 觸發輕微的 `transform: scale(0.99)` 觸覺回饋感。
  - 提供「一鍵重設」按鈕，點擊後觸發淡出動畫並清空 `localStorage`。

#### 3. 網路決策器 (Decision Matrix)
- **結構**：
  - 提供 **「情境導向切換按鈕 (Scenario Tabs)」**：[ 隨興短途旅人 ] / [ 重度 Google/Line 使用者 ] / [ 長途商務與深度預約者 ]。
  - 點擊不同情境，下方比較表 (Table) 的對應方案行 (Row) 會以 `--color-accent` (銅金) 邊框高亮，其餘方案行則淡化至 `opacity: 0.5`，幫助使用者瞬間鎖定方案。
  - 比較表在 Mobile 裝置下自動轉化為三張「滑動式卡片 (Swipeable Cards)」，並附帶橫向滑動指示指示燈 (Dots)。

#### 4. 三層支付防線卡 (Payment Deck)
- **視覺呈現**：
  - 三張卡片疊加排列 (支付寶、微信、實體備援)。
  - 當滾動到此區塊時，卡片以 **「發牌展開 (Deck Deal)」** 動畫平滑滑開，依序由左至右滑入。
  - 卡片本身具有擬真信用卡的漸層微光與圓角，表面有極淡的幾何晶片線條裝飾，散發極簡科技感。
- **付款失敗排查器**：
  - 點擊「付款失敗怎麼辦？」按鈕，卡片平滑翻轉 (Card Flip, 3D CSS rotateY) 顯示背面的「排查四步驟」極簡流程圖。

#### 5. 踩雷避坑區 (Interactive Pitfalls)
- **視覺**：
  - 8 個雷點以灰色警示卡片呈現，封面寫著常見誤區 (例如：「以為 Google Maps 在那邊很好用」)。
  - 右下角有一個硃砂紅的「雷」字圓形印章。
- **互動**：
  - 點擊卡片時，卡片「向下滑出 (Slide down reveal)」或「3D 翻轉」，背景變為 `--color-primary-light`，展示正確的生存解法與官方查核依據，字體高亮為 `--color-primary`。

#### 6. SOS 緊急備援卡 (隨身救援包)
- **視覺**：
  - 整區使用粗紅褐色邊框包圍，左上角有紅色的「急」字印章 Icon。
  - 提供「點擊複製緊急電話/地址」按鈕，點擊後按鈕文字變為「已複製 ✓」並亮起竹青綠 1.5 秒。
- **隨身小卡印製功能 (Print Optimization)**：
  - 提供「產生隨身旅行小卡」按鈕。
  - 點擊時，利用專屬的 CSS `@media print` 樣式，將所有緊急避險資訊、常用電話、飯店中英文地址、手動 Checklist 重新編排為一張 **「三折頁信用卡大小」** 的純文字黑白排版。
  - 網頁其餘的 Hero、Navigator、App 圖示、裝飾性網格、按鈕等在列印時皆會自動隱藏 (`display: none`)，確保使用者在紙質備援列印時不會浪費油墨，且資訊無縫容納於單張 A4 內。


## 6. 技術實作建議

目前工作目錄不是 Git repository 且沒有既有專案檔案。實作前先確認要在此目錄初始化 Git，或從 `https://github.com/kekukeku/mainland.git` clone／設定 remote；不得在未確認遠端既有內容前強制推送。

若遠端沒有既定技術棧，建議採用：

- Vite + React + TypeScript。
- CSS Modules 或單一結構清楚的全域 CSS；不為此頁引入重型 UI framework。
- 內容集中於 typed data objects，政策與來源資訊不要散落在 component JSX。
- 無後端、無帳號、無追蹤 cookie；Checklist 使用 `localStorage`。
- 靜態部署相容（GitHub Pages 或其他 static host）。若使用 GitHub Pages，設定正確的 Vite `base` 路徑。

建議檔案結構：

```text
src/
  components/
    Hero.tsx
    CountdownChecklist.tsx
    DecisionTable.tsx
    AppToolkit.tsx
    RouteCards.tsx
    EmergencyCard.tsx
    SourceDrawer.tsx
  content/
    guide.ts
    sources.ts
  styles/
    tokens.css
    global.css
  App.tsx
  main.tsx
public/
  og-image.webp
  favicon.svg
```

## 7. SEO、無障礙與效能

- HTML `lang="zh-Hant-TW"`，唯一 H1，段落使用語意化 H2/H3。
- Title 建議：「第一次去大陸旅遊攻略｜臺灣人行前準備一頁搞懂」。
- Description 涵蓋台胞證、支付寶、微信、eSIM、住宿、高鐵與必備 App。
- 加入 canonical、Open Graph、基本 `Article` 或 `WebPage` JSON-LD；日期須為真實查核日。
- 鍵盤可操作、清楚 focus、色彩對比達 WCAG 2.2 AA、accordion 有正確 ARIA。
- 360 px 手機寬度不可水平溢出；核心內容在關閉 JavaScript 時仍至少可閱讀。
- 首屏避免大型影片與第三方 embed；圖片使用 WebP／AVIF、尺寸屬性與 lazy loading。
- Lighthouse 目標：Performance、Accessibility、Best Practices、SEO 各 90+。

## 8. 內容查核工作流（實作前必做）

1. 逐條建立 claim inventory：台胞證、首來族、入境、支付費率、網路限制、行動電源、高鐵、住宿、保險與緊急聯絡。
2. 每項時效 claim 至少找到一個主管機關或服務商官方來源。
3. 對無法由官方確認的旅人經驗，標為「經驗建議」，不用肯定句包裝成規定。
4. `sources.ts` 至少保存：`title`、`url`、`publisher`、`sourceType`、`checkedAt`、`supportsSections`。
5. 每次更新政策內容，同步更新頁尾日期與 changelog。
